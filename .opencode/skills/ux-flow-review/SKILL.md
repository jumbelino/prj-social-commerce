# What I do
- inspect a flow from the user's perspective, not the implementation's perspective
- identify friction, ambiguity, and avoidable cognitive load
- prioritize completion, confidence, and recoverability
- focus on practical improvements with low implementation risk
- distinguish core flow blockers from nice-to-have polish

# How to work
1. Identify:
   - who the user is
   - what they are trying to accomplish
   - what success looks like
2. Map the flow into ordered steps.
3. At each step, inspect:
   - what the user must understand
   - what choice they must make
   - what can go wrong
   - how the system helps or confuses
4. Flag common ux problems:
   - unclear next step
   - duplicated effort
   - too many decisions at once
   - hidden consequences
   - weak feedback after action
   - poor error recovery
   - uncertainty about completion
5. Prefer reducing friction before adding more information.
6. Look for moments where the interface makes the user stop and think unnecessarily.
7. Suggest improvements in order of impact on completion rate and user confidence.

# Decision rules
- optimize for task completion, not interface cleverness
- reduce branching unless the branching truly matters
- users should know where they are, what happens next, and how to recover
- if a step exists only because of system constraints, try to hide or soften that burden
